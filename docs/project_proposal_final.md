# InterviewEcho 项目详细方案

---

## 1. 项目概述

InterviewEcho 是一个基于 RAG + 大模型的 AI 模拟面试平台，面向求职者提供贴近真实场景的面试练习与多维能力评估。系统支持三类岗位（Java 后端、Web 前端、Python 算法），覆盖文本与语音双通道作答、GitHub 项目深挖、简历 persona 注入、代码在线判题与自动化评估报告。

---

## 2. 目标与范围

### 2.1 目标
- 提供可个性化配置（岗位/难度/知识点/轮次）的模拟面试。
- 通过 RAG 检索 + LLM 生成实现专业、有深度的追问与打断。
- 输出内容 + 表达多维度量化评估与改进建议。
- 集成自建 Judge0，支持 ACM 模式代码练习与判题。

### 2.2 范围
- 后端：FastAPI 应用，20 个 REST 端点，9 张数据表，4 个外部服务集成。
- 前端：Vue 3 + Vite，含面试房间、报告、代码编辑器（Monaco）、分析图表。
- 评估：四维内容评估 + 三维表达评估 + 代码判题结果。

---

## 3. 系统架构

整体采用前后端分离 + 可移植 RAG 服务 + 外部 AI/判题服务编排，详见 `docs/workflow.md` 架构图。核心链路：

1. 认证（JWT）→ 2. 选岗与配置 → 3. RAG + LLM 面试交互（文本/语音/追问/打断）→ 4. 结束面试触发自动评估 → 5. 报告展示。
代码练习为并行子系统：题库 → 样例运行 / 提交判题（Judge0）→ 提交历史与进度。

---

## 4. 功能模块

| 模块 | 关键能力 | 主要负责人 |
| ---- | -------- | ---------- |
| 认证 | 注册/登录/Token | 后端 |
| 面试交互 | 启动、问答、语音、追问、打断、简历/项目注入 | 后端 + 评估 |
| 评估 | 四维内容评估 + 三维表达评估 + 总分汇总 | 夏浩博 |
| 代码练习 | 题库、样例运行、判题、提交历史（Judge0） | 夏浩博 |
| 知识库/RAG | 题库管理、向量检索 | 关镜文 |
| 前端 | 面试房间、报告、编辑器、图表 | 黄弋涵 |

## 5. 规模度量（FPA）

对当前已交付系统进行 IFPUG 功能点分析，结果如下（详见 `docs/fp_analysis_fpa.md`）：

| 指标 | 数值 |
| ---- | ---- |
| ILF | 9 个 / 69 FP |
| EIF | 4 个 / 22 FP |
| EI  | 6 个 / 30 FP |
| EO  | 5 个 / 26 FP |
| EQ  | 9 个 / 32 FP |
| **UFP** | **179** |
| VAF | 1.10（TDI=45） |
| **AFP** | **≈ 197 FP** |

规模主要集中在事务功能（占 UFP 约 49%），反映系统“交互密集 + AI 处理密集”的特征。

---

## 6. 工作量基线（基于 FPA）

基于 FPA 结果（AFP ≈ 197 FP），按行业经验生产率换算工作量量级：

| 规模 | 生产率假设 | 估算工作量 |
| ---- | ---------- | ---------- |
| 197 FP | 8 人时/FP（偏保守，含 AI 集成复杂度） | ≈ 1576 人时 |
| 197 FP | 6 人时/FP（中等） | ≈ 1182 人时 |

**建议工作量基线 ≈ 1500 人时**，预留 10~15% 缓冲应对 AI 调优等不确定性。

---

## 7. 评估维度对外说明

面向用户的评估维度解读见 `docs/user_manual.md`（评估维度章节）：内容维度（技术正确性 / 业务场景 / 问题解决）+ 表达维度（语速 / 清晰度 / 自信度）+ 总分汇总。

---

## 8. 风险与后续

- AI 评分稳定性：依赖 LLM 输出格式，已通过固定 JSON 约束 + 兜底处理降低风险。
- 判题延迟：C++ 编译延迟较高，后续调优 Judge0 worker 资源。
- 外部服务可用性：LLM/Judge0/GitHub 不可用时已有降级与错误提示。
- 后续聚焦：部署演示环境、性能调优、答辩材料与演示视频。
